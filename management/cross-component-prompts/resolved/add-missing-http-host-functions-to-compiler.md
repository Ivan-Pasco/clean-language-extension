## Solution Implemented

The fix was implemented in compiler version 0.30.7+ with the following changes:

### 1. Added `externals` field to `PluginExpansion` struct

**File:** `src/plugins/mod.rs`

```rust
pub struct PluginExpansion {
    pub statements: Vec<Statement>,
    pub start_function: Option<crate::ast::Function>,
    pub functions: Vec<crate::ast::Function>,
    pub externals: Vec<crate::ast::ExternalFunction>,  // NEW
}
```

### 2. Updated `PluginExpander` to capture and merge externals

**File:** `src/plugins/expander.rs`

- Added `pending_externals: Vec<ExternalFunction>` field to track externals from plugin expansions
- Modified `expand_statements_full()` to capture externals from each expansion
- Modified `expand_program()` to merge pending externals into the AST (with deduplication by name)

### 3. Added Stage 2.6: Bridge Function Conversion

**File:** `src/lib.rs`

After plugin expansion, bridge functions from `plugin.toml` are now converted to `ExternalFunction` declarations and merged into the AST:

```rust
// Stage 2.6: Convert plugin bridge functions to external declarations
let bridge_functions = registry.bridge_functions();
for bf in bridge_functions {
    // Skip if already declared
    if ast.externals.iter().any(|e| e.name == bf.name) { continue; }

    // Convert BridgeFunction to ExternalFunction
    let external_fn = ExternalFunction {
        name: bf.name.clone(),
        parameters: bf.params.iter().enumerate()
            .map(|(i, t)| Parameter { name: format!("arg{}", i), type_: parse_bridge_type(t) })
            .collect(),
        return_type: parse_bridge_type(&bf.returns),
        module: bf.module.clone(),
        location: None,
    };
    ast.externals.push(external_fn);
}
```

### 4. Updated WASM adapter to include externals

**File:** `src/plugins/wasm_adapter.rs`

The `expand_block_internal()` function now includes externals from parsed plugin output in the `PluginExpansion` result.

## Problem (Original)

Compiler v0.31.0 added `external:` block support, but it only works for external declarations in the **source file**. External declarations generated by **plugin expansion** are not registered in the symbol table.

### Working Case (external: in source file)

```clean
external:
    string _req_body_field(string field_name)

functions:
    string test()
        return _req_body_field("email")

start()
    printl(test())
```

This compiles successfully.

### Failing Case (external: from plugin expansion)

Source file (`main.cln`):
```clean
plugins:
    frame.httpserver

endpoints Api:
    POST "/api/register":
        body:
            email: string
        handle:
            return json:
                email: email
```

Plugin generates code including:
```clean
external:
    string _http_respond(integer status, string content_type, string body)
    string _req_body_field(string field_name)
    ...

functions:
    string handler()
        string email = _req_body_field("email")
        return _http_respond(200, "application/json", email)
```

Error:
```
[Plugin Debug] Direct parse succeeded
error: Validation Error
Function '_req_body_field' not found
```

## Root Cause

The compiler parses plugin output successfully ("Direct parse succeeded") but doesn't register `external:` declarations from the plugin expansion into the symbol table before validation.

The validation phase looks for `_req_body_field` but it's not registered, so it fails.

## Expected Behavior

External declarations from plugin expansion should be merged into the symbol table the same way as external declarations from the source file.
