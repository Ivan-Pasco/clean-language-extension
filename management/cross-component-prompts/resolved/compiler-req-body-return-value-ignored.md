Component: clean-language-compiler
Issue Type: bug
Priority: critical
Source Component: Web Site Clean + clean-server

## Summary

The WASM code generated by `cln compile --plugins` does not properly use the i32 return value of `_req_body()`. The clean-server bridge function correctly writes the request body into WASM memory and returns a valid pointer (i32), but the compiled WASM code ignores it — the local variable always reads as empty string.

## Server-side proof (instrumented clean-server v1.9.0)

We added `eprintln!` debug lines to the `_req_body` bridge function in clean-server:

```
[handle_request] POST /api/v1/reports body_len=244     ← Axum receives 244-byte HTTP body
[_req_body] has_context=true, body_len=244              ← RequestContext has the body
[_req_body] memory[ptr..+8]=[244,0,0,0,123,34,114,101] ← Written: len=244, data starts with {"re
[_req_body] __heap_ptr after write=62680                ← Heap pointer advanced correctly
[_req_body] returning ptr=62432                         ← Valid pointer returned to WASM
```

The bridge:
1. Receives the body from RequestContext (244 bytes)
2. Calls WASM `malloc(248)` which returns ptr=62432
3. Writes 4-byte length prefix (244 as u32 LE) at ptr
4. Writes 244 bytes of body data at ptr+4
5. Returns ptr=62432 as i32 to the WASM caller

But the WASM code evaluates `body == ""` as TRUE.

## Minimal reproduction

```clean
plugins:
	frame.httpserver
	frame.data

endpoints server:
	POST "/test" :
		return test_body()

	GET "/health" :
		return test_health()

functions:
	string test_body()
		string body = req.body()
		if body == ""
			return http.respond(200, "application/json", "{\"empty\":true}")
		return http.respond(200, "application/json", "{\"empty\":false}")

	string test_health()
		return http.respond(200, "application/json", "{\"ok\":true}")
```

```bash
cln compile --plugins -o test.wasm test.cln
# Start with any clean-server version >= 1.8.0
clean-server test.wasm --port 3001 --database "..."
curl -d '{"hello":"world"}' -H "Content-Type: application/json" http://localhost:3001/test
```

Expected: `{"empty":false}`
Actual: `{"empty":true}`

## What works vs what fails

| Bridge function | Signature (WASM) | Works? |
|---|---|---|
| `_db_query` | `(i32, i32, i32, i32) → i32` | Yes |
| `_http_respond` | `(i32, i32, i32, i32, i32, i32) → i32` | Yes |
| `_json_encode` | `(i32, i32) → i32` | Yes |
| `_req_ip` | `() → i32` | Yes |
| `_req_body` | `() → i32` | **No** |
| `_req_query` | `(i32, i32) → i32` | **Partial** — some param names work, others don't |
| `_req_header` | `(i32, i32) → i32` | Untested, likely affected |
| `_req_param` | `(i32, i32) → i32` | Untested, likely affected |

Note: `_req_ip()` also takes no parameters but WORKS. So it's not simply "zero-param functions fail."

The pattern may be: functions whose return value is the PRIMARY output fail, while functions where the return value is a side effect of parameter processing work. Or the issue may be in how `endpoints server:` plugin-generated handler wrappers consume return values.

## WASM import verified

```
func[40] sig=9 <req.body> <- env._req_body
```

The import IS correctly linked. The function IS called (debug output proves it). The memory IS written. The pointer IS returned. The WASM code just doesn't use it.

## Where to look in the compiler

The bug is in WASM codegen — specifically how the compiler generates code to handle the i32 return value from imported bridge functions and store it as a Clean Language string variable. The generated WASM likely does one of:

1. Calls `_req_body` but drops the return value from the stack (missing `local.set`)
2. Stores the return value but reads from a different local slot
3. The `endpoints server:` plugin wrapper interferes with the return value

To diagnose: compile the minimal repro above with `--show-ast` or decompile the WASM output with `wasm2wat` and look at the `__route_handler_0` function to see what happens after `call $_req_body`.

## Environment

- Compiler: v0.30.22 AND v0.30.30 — both affected
- Plugins: frame.httpserver 2.6.1, frame.data 2.6.1, frame.ui 2.6.1
- Server: clean-server v1.8.11 AND v1.9.0 — both show correct bridge behavior
- OS: macOS arm64 (dev) and Linux x86_64 (production) — both affected
- Previous working setup: pre-`endpoints server:` code using `start:` block with `_http_route()` on clean-server v1.7.11

## Impact

All POST endpoints on the Clean Language website are non-functional. GET endpoints with `req.query()` are partially broken. This blocks the error reporting API v2 deployment and breaks the existing error reporting API.
