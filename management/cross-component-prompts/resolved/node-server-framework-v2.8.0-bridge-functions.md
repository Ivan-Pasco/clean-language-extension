Description: Implement 8 new server-side bridge functions for Frame Framework v2.8.0

Frame Framework v2.8.0 introduced new plugin features that require 8 new bridge function implementations in the node server. These are all server-side functions — the browser-side UI/canvas functions are handled by the client runtime.

## 1. Database Configuration

### `_db_configure(engine, host, port, database, user, password, pool_max, pool_idle_timeout_ms) -> integer`

**Plugin**: frame.data
**Declaration**: `plugins/frame.data/plugin.toml` — `expand_strings = true`
**Parameters**: 8 (6 strings via ptr+len, 2 integers)

**What it does**: Configures the database connection. Called once at app startup from a `data:` config block.

**Generated by**:
```clean
data:
    engine = "postgres"
    host = "localhost"
    port = 5432
    database = "myapp"
    user = "admin"
    password = env("DB_PASSWORD")
    pool:
        max = 10
        idleTimeout = 30000
```

**Implementation**:
```javascript
function _db_configure(engine, host, port, database, user, password, pool_max, pool_idle_timeout_ms) {
    // Store connection config
    // Initialize connection pool with max connections and idle timeout
    // Supported engines: "postgres", "mysql", "sqlite"
    // Return 0 on success, -1 on error
}
```

**Notes**:
- `password` may be a raw string OR come from `_env_get()` (already resolved by the time this is called)
- If pool already exists, reconfigure it
- Default engine is "sqlite" for local development

## 2. Migration System (5 functions)

### `_db_register_migration(name) -> integer`

**Plugin**: frame.data
**Parameters**: 1 string (migration name, e.g., "001_create_users")

**What it does**: Registers a migration for version tracking. Called at module load time for each `migrate` block in the app.

**Implementation**:
```javascript
function _db_register_migration(name) {
    // Store migration name in an ordered list
    // The migration's up/down functions are resolved by naming convention:
    //   __migration_{clean_name}_up() and __migration_{clean_name}_down()
    //   where clean_name = name with - . and spaces replaced by _
    // Return 1 on success
}
```

### `_db_migration_diff(table_name, field_definitions) -> string`

**Plugin**: frame.data
**Parameters**: 2 strings

**What it does**: Compares declared model schema against the live database table. Returns ALTER TABLE SQL if the table needs updating, or empty string if in sync.

**Implementation**:
```javascript
function _db_migration_diff(table_name, field_definitions) {
    // 1. Query the database for the current table schema:
    //    - PostgreSQL: SELECT column_name, data_type FROM information_schema.columns WHERE table_name = ?
    //    - SQLite: PRAGMA table_info(table_name)
    //    - MySQL: DESCRIBE table_name
    //
    // 2. Parse field_definitions (comma-separated: "name TEXT, email TEXT UNIQUE, age INTEGER")
    //
    // 3. Compare:
    //    - New columns in field_definitions not in DB → ALTER TABLE ADD COLUMN
    //    - Columns in DB not in field_definitions → ignore (don't drop)
    //    - Type mismatches → ALTER TABLE ALTER COLUMN (postgres) or ignore (sqlite)
    //
    // 4. Return the ALTER TABLE SQL string, or "" if no changes needed
    //
    // Example return: "ALTER TABLE users ADD COLUMN phone TEXT; ALTER TABLE users ADD COLUMN age INTEGER"
}
```

### `_db_run_migrations() -> integer`

**Plugin**: frame.data
**Parameters**: none

**What it does**: Runs all pending registered migrations in order.

**Implementation**:
```javascript
function _db_run_migrations() {
    // 1. Ensure migrations tracking table exists:
    //    CREATE TABLE IF NOT EXISTS _migrations (
    //        name TEXT PRIMARY KEY,
    //        applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    //    )
    //
    // 2. Get list of already-applied migrations:
    //    SELECT name FROM _migrations ORDER BY name
    //
    // 3. For each registered migration not yet applied:
    //    a. Call the migration's up function (WASM export: __migration_{name}_up)
    //    b. Record in _migrations table:
    //       INSERT INTO _migrations (name) VALUES (?)
    //
    // 4. Return count of migrations applied
}
```

### `_db_rollback_migration(name) -> integer`

**Plugin**: frame.data
**Parameters**: 1 string (migration name)

**What it does**: Rolls back a specific migration.

**Implementation**:
```javascript
function _db_rollback_migration(name) {
    // 1. Check if migration is in _migrations table
    // 2. Call the migration's down function (WASM export: __migration_{name}_down)
    // 3. Remove from _migrations table:
    //    DELETE FROM _migrations WHERE name = ?
    // 4. Return 1 on success, 0 if migration was not applied
}
```

### `_db_migration_status() -> string`

**Plugin**: frame.data
**Parameters**: none

**What it does**: Returns JSON array of all migration statuses.

**Implementation**:
```javascript
function _db_migration_status() {
```

## 3. UI Layout Loading

### `_ui_load_layout(layout_name) -> string`

**Plugin**: frame.ui
**Parameters**: 1 string (layout name, e.g., "main")

**What it does**: Reads a layout HTML file from `app/layouts/{name}.html` and returns its content. Used by SSR page rendering to wrap page content in a layout template.

**Implementation**:
```javascript
function _ui_load_layout(layout_name) {
    // 1. Resolve path: {project_root}/app/layouts/{layout_name}.html
    // 2. Read file contents
    // 3. Return the HTML string
    // 4. If file doesn't exist, return ""
    //
    // The returned HTML contains <slot></slot> and <slot name="X"></slot>
    // markers that the plugin's generated code replaces with page content.
    // The server just reads and returns the file — slot replacement happens
    // in the compiled WASM code.
}
```

**Example layout file** (`app/layouts/main.html`):
```html
<!DOCTYPE html>
<html>
<head>
    <meta charset="UTF-8">
    <title>{title} - MyApp</title>
    <link rel="stylesheet" href="/css/main.css">
</head>
<body>
    <app-header></app-header>
    <main><slot></slot></main>
    <app-footer></app-footer>
</body>
</html>
```

## 4. UI CSS Injection

### `_ui_inject_head_css(css_string) -> integer`

**Plugin**: frame.ui
**Parameters**: 1 string (CSS content)

**What it does**: Injects a `<style>` block into the page's `<head>` during SSR. Used by the theming system to inject CSS variables.

**Implementation**:
```javascript
function _ui_inject_head_css(css_string) {
    // 1. Store the CSS string in the current request context
    // 2. When rendering the final HTML response, inject:
    //    <style>{css_string}</style>
    //    into the <head> section (before </head>)
    // 3. Return 1 on success
    //
    // If multiple calls are made, concatenate all CSS strings.
    //
    // The generated CSS looks like:
    //    :root { --color-primary: #2563eb; --spacing-sm: 0.5rem; }
}
```

**Integration with SSR pipeline**: The server's HTML response builder should check for injected CSS before sending the response. If CSS was injected via this function, insert a `<style>` tag in the `<head>`.

## WASM Signatures

All string parameters use the `(ptr: i32, len: i32)` convention. Return strings use length-prefixed format.

```wat
;; Database
(import "env" "_db_configure" (func (param i32 i32 i32 i32 i32 i32 i32 i32 i32 i32 i32 i32 i32 i32) (result i32)))
(import "env" "_db_register_migration" (func (param i32 i32) (result i32)))
(import "env" "_db_migration_diff" (func (param i32 i32 i32 i32) (result i32)))
(import "env" "_db_run_migrations" (func (result i32)))
(import "env" "_db_rollback_migration" (func (param i32 i32) (result i32)))
(import "env" "_db_migration_status" (func (result i32)))

;; UI
(import "env" "_ui_load_layout" (func (param i32 i32) (result i32)))
(import "env" "_ui_inject_head_css" (func (param i32 i32) (result i32)))
```

Note: String params with `expand_strings = true` in plugin.toml are passed as `(ptr: i32, len: i32)` pairs, so each string parameter becomes 2 i32 WASM params.

## Context

Frame Framework v2.8.0 added these features:
- Database configuration block (`data:` with engine/host/port/pool settings)
- Migration system (`migrate` blocks with up/down SQL, version tracking, schema diff)
- Layout system (pages reference layouts via `<page layout="main">`, layouts use `<slot>` for content projection)
- Theming system (`ui:` block with `theme:` sub-block generates CSS variables injected into `<head>`)

The clean-server (Rust) is the primary server. Check if these functions are already implemented there. If so, the node server should match the same behavior.

## Files Affected
- Host bridge function registry (wherever bridge functions are registered)
- Database module (for migration system)
- SSR/rendering module (for layout loading and CSS injection)
