Component: clean-language-compiler
Issue Type: bug
Priority: critical
Description: `this` keyword not recognized in class methods generated by plugin expansion

## Reproduction

```clean
plugins:
	frame.data

data User
	integer id : pk, auto
	string email : unique
	string name
	boolean active = true

start:
	printl("App started")
```

Compile with: `cln compile app.cln -o app.wasm --verbose`

## What Happens

1. Plugin expansion succeeds — `Direct parse succeeded`, `classes=1`
2. The expanded class is correctly parsed into HIR
3. Semantic validation fails: `Variable 'this' not found`

## Expected Behavior

The `this` keyword should be valid inside class methods generated by plugin expansion, just like in manually-written classes.

## Plugin Output (abbreviated)

```clean
class User
	integer id
	string email
	string name
	boolean active

	functions:
		boolean save()
			if this.id == 0
				string sql = "INSERT INTO user (...) VALUES (...) RETURNING id"
				string params = __to_params()
				string result = _db_query(sql, params)
				this.id = __extract_id(result)
				return this.id > 0
			...

		boolean delete()
			if this.id == 0
				return false
			...
```

## Root Cause Hypothesis

The semantic analyzer likely processes plugin-generated classes differently from user-defined classes. When a class is parsed from plugin output, `this` may not be added to the scope of its method bodies. The class context (which provides `this`) may be lost during the plugin output → AST merge process.

## Where to Look

- Semantic analysis: scope resolution for `this` inside class methods
- Plugin output merging: how parsed classes from plugin output are integrated into the main program AST
- Check if user-defined classes with `this` work (they should) — compare the AST path for user classes vs plugin-generated classes

## Files Affected
- `src/semantic/` — scope management for class methods
- Wherever plugin-expanded HIR/AST is merged into the main program
